# @gob/bot — Telegram-бот

Ещё один потребитель игрового ядра (`@gob/rules` + `@gob/db` через `@gob/core`).
Ходит к той же БД напрямую (путь A, см. `TODO.md` п.9), без отдельного публичного API.
Технология — [grammY](https://grammy.dev), запуск через long-polling.

> Все инварианты движка обязательны (см. «Золотые правила» в `CLAUDE.md`): игровая
> математика только из `@gob/rules`, никакого clamp/блокирующей валидации, права по `ownerId`.

## Запуск (локально)

1. Создай `apps/bot/.env` (файл в `.gitignore`, в репозиторий не попадает):

   ```dotenv
   BOT_TOKEN="<токен от @BotFather>"
   BOT_HEALTHCHECK_PORT="3001"   # необязательно, по умолчанию 3001
   ```

2. Из корня монорепо:

   ```bash
   pnpm --filter @gob/bot dev    # tsx watch (автоперезапуск)
   # или
   pnpm --filter @gob/bot start  # однократный запуск
   ```

   `pnpm dev` в корне поднимает бота вместе с вебом (turbo).

## Healthcheck

long-polling не открывает свой HTTP-порт, поэтому отдельный лёгкий сервер отдаёт
`GET /health` → `200 {"status":"ok"}` (порт `BOT_HEALTHCHECK_PORT`). Используется
healthcheck'ом сервиса `bot` в `docker-compose.yml` (шаг «Деплой», TODO п.9).

## Команды

Все игровые команды работают над **активным листом** игрока — самым свежим не-NPC-персонажем
владельца (общий фильтр `activeCharacterWhere` в `@gob/core`). Вся математика — из `@gob/rules`,
права — по `ownerId`, никакого clamp (§ золотые правила 1, 2, 6).

| Команда | Что делает |
|---|---|
| `/start <код>` | Привязать аккаунт (код из настроек на сайте → `TelegramLink`) |
| `/me` | Сводка листа: HP/мана/ОД `cur/max`, сытость, характеристики |
| `/hp ±N` `/mana ±N` `/ap ±N` | Изменить живое значение на дельту (+ инлайн-кнопки −5/−1/+1/+5); без аргумента — показать текущее |
| `/bag` | Инвентарь: надетое снаряжение + рюкзак |
| `/money` | Баланс: золото/серебро/бронза + итог в бронзе |
| `/roll …` | Броски кубов (см. ниже) |
| `/ping` | `pong` (проверка живости) |

### `/roll`

Структура: `/roll <обозначение> <кубы>` либо просто `/roll <кубы>`. Кубы — `NdM` / `dM` / `M`
(один куб); принимается и кириллическая `д`. При нескольких кубах выделяются макс 🔺 и мин 🔻
(бросок с преимуществом/помехой — нужный берёт сам игрок). Обычный бросок работает без привязки.

| Пример | Что делает |
|---|---|
| `/roll 20` · `/roll 3d6` | Обычный бросок (сумма + мин/макс), без семантики |
| `/roll STR 12` | Проверка характеристики (`STR DEX INT SPI END LUC`): `1` — крит-успех, `≤` значения характеристики — успех, макс грань — крит-провал; меньше лучше |
| `/roll ATK 2d20` | Попадание (`≥ грань−1−ЛОВ`) + крит (`≥ грань−критМод`, критМод=⌊УДЧ/lucCritStep⌋). Крит без попадания = критический промах. Натуральная 1 = антикрит (попадание по союзнику; урон кидается как обычное попадание, без крит-кубов) |
| `/roll DMG 3d6` | Сумма кубов + плоский доп. урон от характеристики надетого оружия (не входит в сумму) |
| `/roll BBL` | Проверка Бабла по d100: пишет состояние (спал → заряды 0; устоял → заряды−1, мин 1); индикатор «дух ≥ 6 → не остаётся на 0» |

Read-модели (`getActorCharacterSummary`/`getActorMoney`/`getActorInventory`/`getActorRollContext`)
и оркестровка правок (`adjustRuntimeResource`/`applyActorBubbleRoll`) живут в `@gob/core`; парсинг
команды и RNG — `src/roll.ts`; форматирование сообщений — чистые функции `src/format.ts`. Всё
покрыто тестами.
